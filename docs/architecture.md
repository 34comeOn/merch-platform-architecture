## 1. Цели и задачи (Goals & Scope)
**Цель проекта**: Создание доверенной B2B-экосистемы для заказа корпоративного мерча, сокращающей путь от идеи до готового продукта с недель до минут.

* **Ключевые функциональные требования**:
* **Inspiration Feed**: Визуальный поиск идей на базе реальных кейсов.
* **AI Design Lab**: Генерация макетов «на лету» (YandexART + Sharp).
* **Reverse Auction**: Автоматический подбор исполнителей (модель Яндекс такси).
* **Trust & Safety**: Система арбитража и безопасных платежей.

**Нефункциональные требования**:
* **Availability**: 99.9% (критично для B2B).
* **Scalability**: Обработка до 1000 RPS на чтение каталога и до 100 параллельных генераций макетов.
* **Latency**: Отображение AI-превью < 3 сек.


## 2. Общая архитектурная стратегия
Система строится на принципах **Cloud Native** и **Microfrontends**, чтобы обеспечить независимую разработку разных частей платформы (каталог, конструктор, кабинет поставщика).

**Ключевые решения**:
* **Backend**: Микросервисная архитектура на Node.js/Go.
* **Frontend**: React + Module Federation для разделения зон ответственности.
* **Communication**:
    * *External*: REST/GraphQL для фронтенда.
    * *Internal*: gRPC для межсервисного взаимодействия и Kafka для асинхронных задач (генерация макетов, рассылка заказов).
* **Data Consistency**: Принцип Eventual Consistency для распределенных систем (заказ считается созданным, когда он попал в очередь, даже если AI еще не дорисовал превью).


## 3. Высокоуровневая архитектура (HLD)
### Слой микросервисов (Back-end)
В дополнение к базовым сервисам (Image Processor, AI Wrapper) внедряются узлы управления жизненным циклом заказа:

* **Matching Engine (Диспетчер)**:
    * **Назначение**: Оркестрация обратного аукциона между заказчиком и исполнителями.
    * **Логика**: При поступлении заказа сервис ищет подходящие производства в реальном времени.
    * **Оптимизация**: Использование **Гео-индексов** (по аналогии с Uber/Яндекс Такси) для минимизации логистических издержек. Приоритет отдается исполнителям, находящимся в том же регионе, что и точка доставки.
    * **Ранжирование**: Учитывает не только цену, но и текущий рейтинг («Verified by Yandex») и загрузку мощностей (Capacity).
* **Supplier App (MFE)**:
    * **Тип**: Микрофронтенд (отдельный инстанс).
    * **Функционал**: Персональный кабинет для производственных цехов.
    * **Управление очередью**: Просмотр входящих тендеров, принятие заказов в один клик, отслеживание этапов производства (печать, упаковка, отгрузка).
    * **Ресурсный менеджмент**: Возможность в реальном времени «выключать» цех из поиска, если мощности перегружены.


## 4. Схема взаимодействия (Matching Flow)
1. **Event**: Пользователь подтверждает AI-макет → создается событие ORDER_CREATED.
2. **Dispatch**: Matching Engine подписывается на событие и делает выборку из базы поставщиков по параметрам: Гео-позиция, Тип печати, Свободные слоты.
3. **Push**: Система отправляет Push-уведомления в Supplier App подходящим исполнителям.
4. **Accept**: Первый исполнитель (или тендерный отбор), нажавший «Принять», фиксирует сделку.
5. **Status**: Через WebSocket статус заказа обновляется у клиента («Исполнитель найден, производство начато»).


## 5. Арбитраж и Безопасность
* **Escrow Service**: Платеж клиента удерживается системой.
* **Arbitration Layer**: Яндекс выступает гарантом. Если поставщик нарушил сроки или качество (проверка по фото-фиксации в Supplier App), арбитр инициирует возврат средств.


## 6. Расчет ресурсов (Capacity Planning)
Вес одного среднего заказа:

1. **Рабочий макет (Print-ready, TIFF/PDF)**: ~50–150 МБ.
2. **Превью (WebP) для клиента**: ~200 КБ.
3. **Исходный логотип клиента (Vector/PNG)**: ~5 МБ.
4. **Метаданные в БД (JSON)**: ~10 КБ.
Итого на 1 заказ: ~100–150 МБ.


| **Количество** | **Объем макетов (S3)** | **Записи в БД (PostgreSQL)** |
| :--- | :---: | :--- |
| 100 заказов | ~15 ГБ | ~1 МБ |
| 1 000 заказов | ~150 ГБ | ~10 МБ |
| 10 000 заказов | ~1.5 ТБ | ~100 МБ| 

Чтобы S3 не раздувался до бесконечности, внедряем Lifecycle Policy. Макеты для печати автоматически удаляются через 30-60 дней после завершения заказа. В базе остается только история и маленькое превью.

* **Throughput**: 1000+ RPS (чтение), 100+ параллельных AI-генераций.
* **Storage**: ~1.5 ТБ на каждые 10 000 заказов.
* **Matching Engine**: Хранение гео-координат производств в **Redis (GeoJSON)** для мгновенного радиусного поиска (O(log N)).
* **Supplier App**: Трафик минимизирован за счет раздачи тяжелых макетов через **Presigned URLs** напрямую из S3.
